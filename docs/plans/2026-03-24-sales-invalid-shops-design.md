# 销售统计无效店铺功能需求梳理

## 1. 目标

在现有系统中新增一个分类页面，用于统计**每个销售名下的无效店铺数量**，并支持查看对应店铺明细。

该页面的主要价值是：

- 快速识别销售所签店铺的质量情况
- 发现“签约后一个月内抽点金额偏低”的店铺
- 支持按销售维度做质量复盘

---

## 2. 页面位置

在左侧导航中，新增一个分类：

- `销售统计无效店铺`

放置位置：

- 在 `饿了么每日抽点明细` 下方

> 说明：虽然入口位置放在每日抽点明细分类附近，但统计范围是**双平台合并**，不是只看饿了么。

---

## 3. 统计范围

### 3.1 平台范围

按你最新确认，本次统计范围为：

- 美团
- 饿了么

并且：

- **按销售合并统计**
- 不拆分成“美团一组、饿了么一组”

### 3.2 店铺来源

店铺主数据来源：

- `shops`

抽点数据来源：

- `daily_point_details`

---

## 4. 无效店铺判定规则

### 4.1 核心规则

一个店铺是否视为“无效店铺”，按下面规则判断：

```text
从签约日期开始，统计一个月内的抽点总额；
若总额 < 90 元，则视为无效店铺。
```

### 4.2 “一个月”的口径（当前建议）

建议定义为：

- 起点：`contractSignedDate` 当天（含）
- 终点：签约日期往后顺延 1 个月，再减 1 天（含）

例如：

- 签约日期：`2026-03-12`
- 统计区间：`2026-03-12` ~ `2026-04-11`

> 这是当前建议口径。若后续你希望改成“签约日起 30 天”而不是“自然月 1 个月”，实现时也可调整。

### 4.3 抽点总额定义

抽点总额 = 统计窗口内，该店铺全部抽点明细的 `amountValue` 求和。

说明：

- 正数、负数都参与求和
- 最终按净值判断是否 `< 90`

---

## 5. 店铺与抽点明细的匹配规则

为了保证尽量匹配到店铺数据，建议采用以下顺序：

1. 优先用：
   - `merchantId`
   - `storeId`
2. 其次用：
   - `shopName`

也就是说：

- 若抽点明细中的 `storeId / merchantId` 能匹配主表商家 ID，则优先采用 ID 匹配
- 若 ID 匹配不到，再退回到店铺名匹配

## 7. 建议接口设计

建议新增独立 API，而不是复用当前抽点明细列表接口。

### 7.1 汇总 + 明细一体返回

建议新增接口：

- `GET /api/daily-point/invalid-shops-by-sales`

建议参数：

- `page`
- `pageSize`
- `salesName`（可选）
- `keyword`（可选，店铺名/商家 ID）

建议返回结构：

```ts
{
  summary: Array<{
    salesName: string;
    invalidShopCount: number;
    totalAmount: number;
    averageAmount: number;
  }>;
  data: Array<{
    shopId: string;
    shopName: string;
    merchantId: string;
    deliveryPlatform: string;
    city: string;
    salesName: string;
    operatorName: string;
    contractSignedDate: string;
    windowStartDate: string;
    windowEndDate: string;
    totalAmount: number;
    result: "无效店铺";
  }>;
  total: number;
  page: number;
  pageSize: number;
}
```

## 8. 数据计算流程

### 步骤 1：读取店铺主表

从 `shops` 读取全部候选店铺，至少需要字段：

- `_id`
- `shopName`
- `merchantId`
- `deliveryPlatform`
- `city`
- `salesName`
- `operatorName`
- `contractSignedDate`

### 步骤 2：为每个店铺构建统计窗口

对每个店铺生成：

- `windowStartDate = contractSignedDate`
- `windowEndDate = contractSignedDate + 1个月 - 1天`

### 步骤 3：匹配抽点明细

在 `daily_point_details` 中查出落在该窗口内的抽点明细，按店铺聚合求和：

- `sum(amountValue)`

### 步骤 4：判定无效店铺

若：

- `totalAmount < 90`

则计入无效店铺名单。

### 步骤 5：按销售汇总

把无效店铺结果按 `salesName` 分组，得到每个销售的：

- 无效店铺数量
- 总抽点金额
- 平均抽点金额

---

## 9. 边界情况

### 9.1 无签约日期

若某店铺 `contractSignedDate` 缺失：

- 当前建议：跳过，不参与统计

### 9.2 无销售

若 `salesName` 为空：

- 归类为 `未分配`

### 9.3 新签店铺尚未满一个月

这里有两个可选口径：

#### 方案 A：按当前可见窗口统计（推荐）

即使店铺还没满一个月，也统计“从签约日起到当前已有数据的金额”，若仍 `< 90` 就先判为无效。

优点：

- 能尽早预警

缺点：

- 容易把刚签约不久的店过早归为无效

#### 方案 B：必须满一个月才纳入

只有签约日期已经过去满一个月的店，才参与无效判定。

优点：

- 更严格

缺点：

- 预警更晚

> 这个点是当前最需要你确认的业务口径。

### 9.4 一个店有重复主表记录

若同一商家 ID 在 `shops` 中存在多条记录：

- 当前建议按店铺记录逐条计算
- 不自动合并历史合同记录

## 12. 最终确认口径

你已确认采用：

### A. 必须签约满一个月才纳入统计

即：

- 只有当店铺从 `contractSignedDate` 起已经满 1 个月
- 才进入“无效店铺”判定范围

然后再判断：

```text
签约日起一个月内抽点总额 < 90 元
```

则该店铺计为无效店铺。

### 该口径的业务含义

优点：

- 更稳定
- 不会误伤刚签约不久的新店
- 更适合作为销售质量复盘依据

结果：

- 页面中只展示“已满一个月且抽点总额不足 90 元”的店铺
- 尚未满一个月的店铺不会提前进入无效名单
